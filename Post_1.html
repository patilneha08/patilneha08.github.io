Getting Started with RAG: What I Built and What I Learned

NP
Neha Patil · Week 1 · 8 min read

Why I built DocChat AI

When I started my ML coursework at NYU, I kept running into the same problem: I had a pile of research papers and lecture notes, and no good way to query across them. Ctrl+F only goes so far. So I decided to build a document Q&A assistant — and in the process learned way more about RAG than I expected to.

This is a breakdown of what I built, the decisions I made, and the parts that surprised me. If you're building a RAG system for the first time, hopefully this saves you some of the trial and error.

What is RAG, actually?

Retrieval-Augmented Generation means you give an LLM access to external knowledge at inference time — rather than baking it into the weights via fine-tuning. The flow looks like this:

  1. User asks a question.
  2. You embed the question into a vector.
  3. You search a vector store for the most similar document chunks.
  4. You pass those chunks as context to the LLM along with the question.
  5. The LLM generates an answer grounded in the retrieved context.

Simple in theory. The complexity lives in the details — especially steps 2, 3, and 4.

💡 Key insight: The quality of your retrieval directly caps the quality of your generation. A great LLM can't answer from bad context. Fix retrieval first.

The stack I used

For DocChat AI I went fully local: LangChain for orchestration, ChromaDB as the vector store, Ollama with Mistral 7B for generation, and FastAPI + Streamlit for the interface. No OpenAI API key required — everything runs offline. 

# Basic ingestion pipeline
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OllamaEmbeddings
from langchain.vectorstores import Chroma

loader = PyPDFLoader("paper.pdf")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64
)
chunks = splitter.split_documents(docs)

vectorstore = Chroma.from_documents(
    chunks,
    OllamaEmbeddings(model="mistral")
)

The chunking decision matters more than you think

I started with a fixed chunk size of 1000 tokens. The results were mediocre — retrieved chunks would sometimes cut off mid-sentence or miss the context that came right before. Switching to RecursiveCharacterTextSplitter with overlap fixed most of this. The overlap means related content is duplicated across adjacent chunks, so retrieval is less brittle.

What I'd do differently: experiment with semantic chunking (splitting on meaning boundaries rather than token count) — I haven't tried it yet but the results in papers look promising.

What surprised me

The biggest surprise was how much the prompt template affected answer quality — almost as much as retrieval itself. Explicitly telling the model to say "I don't know" when the context doesn't contain the answer dramatically reduced hallucinations.

The second surprise: Mistral 7B is genuinely impressive for a local model. It's not GPT-4, but for factual Q&A over document context it performs well above what I expected.

Next steps

I want to add hybrid search (combining dense vector retrieval with BM25 keyword search) and experiment with reranking retrieved chunks before passing them to the LLM. Both are known to push retrieval quality up significantly.
